## 1. 产品是什么

ticket-hub 是**跨源工单枢纽**：把 KSM、智齿、Zammad、AI 客服 escalation 四个来源的工单统一收进来，用 AI（LLM）自动做「分类 → 路由分配 → 必要时拆单/去重」，主管在需要时做人工修正。工单最终「毕业」为 4 种研发协同出口类型之一（运营 / Bug 修复 / 需求 / 内部任务），推送到 Linear 跟踪研发进度；第 5 种「投诉」类型停在工单层由人工处理，不进入毕业流程。

整个系统的核心心智：**AI 干活，人管例外**。系统会把绝大多数工单自动分类、自动分配、自动毕业；主管的工作集中在「工作台」首页的一张**需人工介入队列**里——凡是 AI 拿不准、失败、或有权限要求人确认的事，都会出现在这张队列里等你处理。

## 2. 登录

访问系统会先看到登录页，标题「登录 ticket-hub」。**唯一登录方式是飞书扫码**——没有账号密码表单。

- 点「飞书扫码登录」跳转飞书扫码授权
- 授权成功后自动创建/更新本地用户档案（姓名、邮箱、手机号从飞书同步），跳回工作台
- 若上次扫码回调失败，页面会用红字提示原因
- Token 有效期 7 天，过期后 API 请求会自动清本地登录态并跳回登录页

新用户首次登录默认角色是「普通成员」，需要管理员到「管理 → 人员与分工」手动升级角色，见 [§9.1](#91-人员与分工)。

## 3. 角色与权限

系统有 5 个角色，按能力递增排列：

| 角色 | 中文名 | 能做什么 |
|---|---|---|
| `member` | 普通成员 | 只能查看工单和仪表板，无管理权限 |
| `assignee` | 处理人 | 可被 AI 自动分配工单，处理名下工单并回复客户 |
| `knowledge_op` | 知识运营 | **反思诊断工作台**全部功能：诊断 AI 客服失败原因、修订对客 AI 客服 skill、核对/回放验证、发布 |
| `supervisor` | 主管 | 工作台队列全部处理权限（拆单/去重/重推 Linear/关闭投诉/重新分配等）+ 人员分工管理（天然涵盖知识运营能力）|
| `admin` | 管理员 | 以上全部 + 用户角色变更、目录管理（产品线/模块/Feature）、**内部编排 Agent** 的 Skill 配置 |

**左侧导航按角色显示**：

| 导航项 | 可见角色 |
|---|---|
| 工作台 / 工单 / 研发协同 | 全部角色 |
| 反思诊断 | knowledge_op / supervisor / admin |
| 管理 | supervisor / admin |

> **注意一个 UI 细节**：「管理」下的「目录管理」「Skill 配置」两个 tab 对 supervisor 也是可见的（导航层面只挡到「管理」这一级），但点击具体操作后端会返回 403「需要 admin 角色」——这两块**实际只有 admin 能操作成功**。人员与分工页 supervisor 可正常使用，只是看不到「分工变更审计」卡片（仅 admin 可见）。

## 4. 工作台（首页）

登录后默认落地页，`/`。分两段：

### 4.1 整体看板

- 顶部可切换统计口径：**今日 / 本周 / 本月**
- **状态漏斗**：接收 → AI 已分类 → 已分配 → 处理中 → 已解决/关闭，五段柱状图，点击任一段跳转工单列表并带对应筛选
- **SLO 指标卡**（4 张）：每张显示当前值 + 环比升降（↑/↓ 多少个百分点）+ 7 日趋势迷你折线（数据不足 2 天时显示「趋势积累中」）
- **来源分布条**：KSM / 智齿 / Zammad / AI客服 四色横向堆叠条 + 图例数字
- 若系统检测到配置缺口（如某模块无人接手、兜底处理人未设置），顶部会出现**黄色配置警告 Banner**，可折叠，点「去设置 →」跳转对应管理页

### 4.2 需人工介入队列

这是主管/知识运营/管理员每天要看的核心区域（仅 supervisor/admin 角色可处理；普通成员/处理人看到的是提示文案，看不到队列本身）。七类事项混排展示，右上角有筛选 chip（全部 / 每个类型 + 数量）：

| 类型 chip | 颜色 | 含义 | 处理方式 |
|---|---|---|---|
| 拆单提案 | 紫 | AI 判定一个工单混着多个独立问题，等你确认要不要拆 | 「执行拆单」按提案拆成子工单；「忽略」维持原单 |
| 重复提案 | 青 | AI 判定这是另一个工单的重复反馈 | 「采纳合并」挂到原始工单的 hub_issue；目标工单还没毕业时按钮会禁用，提示先去创建 |
| Linear 待人工 | 琥珀 | hub_issue 该推 Linear 但推送失败或处理人在 Linear 查无此人 | 「重推」——如果是「查无此人」，先去人员管理做「从 Linear 同步」补映射再重推 |
| 未分配 | 红 | 路由没命中任何分工规则，落进了兜底池 | 「手动分配」跳转工单列表的未分配筛选视图，手动指定处理人 |
| 投诉 | 实心深红 | AI 判定为投诉类，投诉**永远不会自动毕业**，必须人工处理 | 选类型（运营/Bug修复/需求/内部任务）后「转毕业」，或纯情绪投诉直接「关闭」 |
| 待诊断 | 绿(emerald) | AI 客服 escalation 工单还没做反思诊断 | 「去诊断」跳转反思诊断工作台并带上该工单 |
| SLA 通知 | 灰 | 系统 SLA 超时告警（原「主管工作台收件箱」） | 「已确认」标记已读 |

每行显示等待时长（超过一定小时数变琥珀/红色高亮，提示优先处理）。队列清空后会显示「🎉 队列已清空」提示，并引导去「研发协同」看推进中的工单。

## 5. 工单管理

### 5.1 工单列表 `/tickets`

- **筛选栏**：来源（全部/KSM/智齿/Zammad/AI客服）、状态（received/linked/waiting_reply/in_progress/replied/done，状态码为英文原文未译中文）、「仅未分配」勾选框；有筛选时出现「重置筛选」
- **列**：工单号 / 标题 / AI 分类（彩色徽标：运营=琥珀、Bug 修复=玫红、需求=蓝、内部任务=灰、投诉=实心红；未分类显示灰字「未分类」）/ 来源 / 模块 / 处理人（头像+姓名，未分配显示「—」）/ 状态 / 收到时间
- 底部分页

**批量操作**（仅 supervisor/admin 可见）：勾选多行 → 底部浮出操作条「已选 N 条」→「重新触发分配」→ 弹窗二次确认 → 系统按当前分工配置重新计算处理人，结果表格逐条显示「已分配 / 默认池 / 多组匹配 / 未匹配 / 不存在」。

### 5.2 工单详情 `/tickets/:id`

纯只读展示页，**没有编辑或人工分类修正按钮**（分类修正是通过反思诊断的 skill 修订机制，不是逐单改）。包含：

- 基本信息：模块、特性、产品线、负责人、关联 hub_issue（可点进 [研发协同](#7-研发协同hub-工单)）、客户（可点进 [客户详情](#6-客户查询)）、提交人、收到/回复时间
- 原文（工单正文，等宽字体展示）
- 缓存回复（若该单已从 hub_issue 回复级联下来，显示回复版本号内容）
- **知识反哺面板**：只有当该工单是 AI 客服 escalation 单、且知识反哺功能已开启、且你是 supervisor/admin 时才出现，内容与 [反思诊断工作台](#8-反思诊断工作台) 相同，方便直接在工单详情里改 skill
- 变更时间线：状态变化、关联建立/解除记录，逐条显示操作人和原因

### 状态含义速查

`received`（接收）→ AI 分类完成后进 `linked`/按类型分流 → 客户已答复走向 `waiting_reply` → 处理人跟进为 `in_progress` → 回复客户 `replied` → 关闭 `done`。研发类工单的进一步进度体现在挂靠的 hub_issue（详见 [§7](#7-研发协同hub-工单)）。

## 6. 客户查询 `/customers`

### 6.1 搜索

单个搜索框，支持：
- `erp_uid` / 手机号 / 邮箱 —— **精确匹配**
- 姓名 —— 模糊匹配

结果卡片显示姓名、公司、联系方式；若该客户已被合并到另一条记录，显示「已合并 → #目标ID」徽标。

### 6.2 客户详情

- 若客户处在合并链路中，顶部黄色提示框展示完整合并链 `#id → #id → ...`
- **身份映射区**：一个客户可能在多个来源系统留下不同身份记录（KSM/智齿/Zammad/AI客服/Linear），每条身份卡片显示解析方式（按 erp_uid / 手机 / 邮箱 / 来源自定义ID / 人工 / 无匹配）、是否人工确认过、姓名/联系方式字段、首次见到/最近活跃时间
- 本页纯只读，无合并/拆分操作入口（这类操作走后端 API，暂无前端表单）

## 7. 研发协同（Hub 工单）`/hub-issues`

工单被 AI 或主管「毕业」后，会生成一个 `hub_issue`（格式 `HUB-000123`），这是**跨工单**的处理单元——多个来源工单反馈同一个问题，会共用同一个 hub_issue。

### 7.1 四类型分视图

顶部 tab：全部 / 运营 / Bug修复 / 需求 / 内部任务。列表列：工单标题区、状态（研发类显示 Linear 状态徽标+identifier，运营类显示「已回复 v{n}」/「未回复」，内部任务显示飞书任务状态）、停留时长（≥7 天琥珀、≥14 天红色高亮）、催办/客户反馈区、动作列。

### 7.2 运营（Operation）类

在 [工单详情页](#52-工单详情-ticketsid) 之外，运营类 hub_issue 有独立的**回复编辑器**（在 hub_issue 详情页）：撰写/修改面向客户的统一回复 → 保存后自动**版本化**存档 + **级联**到所有关联工单的缓存 + 写入待回写队列（源系统开关打开后自动回写答复给客户）。

### 7.3 Bug 修复 / 需求（研发类）四个协同动作

这四个动作都在 hub-issues 列表页的「动作」列内联触发（部分需先展开弹窗），仅 supervisor/admin 可操作：

| 动作 | 触发条件 | 效果 |
|---|---|---|
| **催办** | 已推 Linear 且未关闭；24 小时内催过会显示「24h 内已催」置灰 | 在对应 Linear issue 下发一条催办评论，计数 +1 |
| **发版通知** | 研发标记完成、待通知客户 | 填「上线版本号」+ 通知文案 → 发给所有关联客户渠道，工单转「待回访」 |
| **记录回访** | 已发过发版通知、状态为「待回访」 | 记「确认解决」或「客户仍报错」+ 回访备注；标「仍报错」会红字高亮，提醒评估是否要升级新工单 |
| **登记自修复 bug**（列表页顶部「＋ 登记自修复 bug」按钮）| 研发自查发现且已修复、无客户来源 | 建一个 standalone Bug_fix hub_issue，标「自查」灰徽标，**不触发任何客户通知**，不推 Linear |

### 7.4 按责任人拆分（owner-split）

当一个 Demand/Bug_fix hub_issue 的工作实际分属多个责任人时，在 hub_issue 详情页可以「按责任人拆分」：

1. 前提：该 hub_issue 必须**已经推送 Linear**（子任务要挂在 Linear 主 issue 下）
2. 填写 2–20 行「子任务标题 + 责任人」表单，点「拆分为 N 个子 issue」
3. 系统在 Linear 下建 N 个子 issue（自动落到责任人所属 Linear team），并在详情页显示「子任务里程碑」列表（每行状态、上线时间）
4. **进度自动通知客户**：每个子任务在 Linear 标记完成，系统自动向客户发一条进度消息「N 个子任务，本次完成第 X 个，剩余 N-X 个处理中」——**不会关闭客户工单**；只有最后一个子任务完成时才发终版通知并关闭工单。这样客户不会因为「还差一个小需求没做完」而工单一直挂着不知道进度
5. 限制：v1 一个 hub_issue 只能拆一次，不支持追加/重拆；某责任人若在 Linear 查无此人会直接报错拒绝，需先去人员管理做「从 Linear 同步」

### 7.5 内部任务（Internal_task）

展示挂接的飞书任务 ID、状态、同步时间，本类型不涉及客户通知，暂无本手册未提及的额外操作。

## 8. 反思诊断工作台 `/reflect`

**仅 AI 客服 escalation 工单**（客户对飞书 AI 客服的回答不满意、转人工的场景）会出现在这里，其他来源工单不适用。仅 knowledge_op / supervisor / admin 可见。

左侧栏是待处理的 escalation 工单列表；点选一个后，右侧是「医生看病」三段式布局：

### 8.1 第一段·看症状 · 诊断

- **黄金三元组**：客户原始问题 / AI 客服原答复 / 客户不满反馈
- 完整多轮会话（可展开）
- AI 答复时引用的知识条目（含相似度分数，低分是「没检索到对的知识」的强信号）
- **人工核对的正确答案**（可编辑填写，作为 AI 反思推断的判断基准，填了推断更准）
- **AI 反思推断**（点按钮触发一次 LLM 调用）：走固定三步排查（客户真正卡点是什么 → 知识库是否覆盖正解 → AI 有没有用好命中的知识），给出病因判定 + 置信度 + 修订建议，可一键「采纳该诊断」把推断结果同步到下面的病因判定
- **病因判定**（多选，一次失败可能是多个病因叠加）：
  - `skill` 问题 —— 引用了对的知识，但答复没用好
  - `knowledge` 问题 —— 引用的知识本身有误或过期
  - `retrieval` 检索缺失 —— 没检索到相关知识
- **知识库核对面板**：按当前工单的失败问题自动检索飞书知识库
  - 命中条目 → 逐条给「在飞书打开」链接，人工核对「知识本身对不对」「AI 有没有用好」
  - **无命中 → 高亮显示「检索缺失」**——这是 retrieval 病因的直接证据
  - 在飞书里改完知识后，点「↻ 重新检索」刷新结果，再用右侧 replay 验证效果
- **修复清单**：勾选的每个病因自动生成一条待办项，全部勾完显示绿色「修复闭环 ✓」徽标。skill 项需右侧走完 replay 验证才建议勾；knowledge/retrieval 项目前是人工去飞书改完后手动勾选确认（暂无自动核验，飞书 KB API 尚未提供发布回调）

### 8.2 第二段·开药 · 修订 skill

只对 skill 病因适用：多文件编辑器修改 AI 客服的对客 skill 文件，改完点「创建 draft」（不影响生产，草稿态）。

> **重要区分**：这里改的是 AI 客服的**对客 skill**（回答客户问题用的），跟「管理 → Skill 配置」里的**内部编排 skill**（工单分类/拆单用的）完全是两回事，互不影响，见 [§9.3](#93-skill-配置内部编排-skill)。

### 8.3 第三段·试药与处方

- **replay**：用「当前生产版」或「刚才创建的 draft」+ 「最新知识库内容」重新回答同一个问题，展示新旧答复对比 + 引用差异
- 满意后点「发布」——draft 转正式生产版本，这次 escalation 的诊断链路闭环

若知识反哺功能未配置（`KNOWLEDGE_FEEDBACK_ENABLED` 未开或凭证缺失），第二、三段会降级为提示条，第一段的诊断和知识库核对不受影响仍可用。

## 9. 管理后台 `/admin/*`

顶部三个 tab：人员与分工 / 目录管理 / Skill 配置。

### 9.1 人员与分工 `/admin/users`

**全局兜底处理人**（页面顶部常驻条）：路由匹配不到任何分工规则时，工单会落到这个人手上兜底；没设置会有醒目警示。

**左列·人员列表**：
- 「↻ 从飞书同步」——弹窗展示部门树，勾选部门/成员批量导入或更新用户档案（姓名/工号/邮箱/手机自动从飞书取，本地已存在的会更新；飞书已停用的账号勾选框会禁用）
- 「↻ 从 Linear 同步」——按邮箱把本地用户和 Linear 工作区成员匹配，补全 `linear_user_id`/`linear_team_id`（无邮箱的组账号会被跳过，弹窗会提示跳过数量）
- 搜索 + 在岗/停用状态筛选

**右列·选中人员档案**（5 个卡片）：
1. 基本信息（只读，来自飞书）
2. **角色权限**——5 张角色卡单选，点击立即生效，见 [§3](#3-角色与权限) 的角色说明
3. **管理范围**——该人负责哪些「产品线+模块」或「跨产品线 Feature 兜底」，AI 路由据此把工单分给他；没有分工的人不会被自动分配工单
4. 外部映射——飞书 UID（必需）、Linear（必需，缺了研发类工单推不过去）、KSM 账号（可选）
5. 危险区——停用/启用账号（停用会立即移出 AI 路由池，需先转派名下进行中工单）

仅 admin 能看到「分工变更审计」卡片，查看历史分工调整记录。

### 9.2 目录管理 `/admin/catalog`

维护「产品线」「模块」「Feature」——**下拉框选项的唯一数据源**，其他所有页面的产品线/模块下拉都从这里读。

- Tab「产品线 / 模块」：新增产品线（code/名称/SLA回复小时数/SLA解决小时数）；每个产品线下可挂多个模块；可编辑 SLA 数值、删除（产品线下有模块时不能直接删，要先删完模块）
- Tab「Feature」：跨产品线的兜底分类（如「数据导入」「权限管理」），新增/停用

> **需要 admin 角色**——supervisor 能看到页面但操作会收到「需要 admin 角色」的 403 提示。

### 9.3 Skill 配置（内部编排 skill）`/admin/skills`

页面顶部有醒目横幅区分：**这里管的是 hub 内部流水线用的 Agent 提示词**（分类 / 拆单判定 / 查重 / 截图OCR / 反思推断），**不是**面向客户的 AI 客服对客话术（那个在 [反思诊断工作台](#8-反思诊断工作台) 改）。

版本管理是「三槽」模式：
- **current** —— 生产在用版本，只读
- **draft** —— 候选草稿，编辑区在这里改
- **previous** —— 上一个版本，用于回滚

标准流程：改 draft → 「① 保存 draft」→「② 差异回放验证」（用最近 8 条真实工单分别跑 current 和 draft 的提示词，对比分类结果有没有变化，逐条列出）→ 确认没问题后填「提升说明」→「③ 提升为 current」（会弹二次确认，**即刻对线上生效**）。也可以在 previous 槽点「回滚到 previous」一键退回上一版。

「从文件导入」按钮用于把代码仓库里的 prompt 文件批量导入数据库初始化。

> **同样需要 admin 角色**才能操作成功。

## 11. 术语表

| 术语 | 含义 |
|---|---|
| ticket（工单） | 单个来源系统（KSM/智齿/Zammad/AI客服）产生的一条客户反馈，是本系统的最小单位 |
| hub_issue | 工单「毕业」后生成的跨工单处理单元，多个工单可挂到同一个 hub_issue（去重后共用） |
| 毕业 | 工单被 AI（达到置信度门槛）或主管手动确认分类后，升级为 hub_issue 的过程 |
| 拆单 | AI 判定一个工单混杂了多个独立问题，物理拆分成多个子工单（Child），各自分类和分配 |
| 触发分类（triage） | 单次 LLM 调用同时完成「这是什么类型」和「是不是混合多问题」的判定 |
| 5 种出口类型 | 运营（Operation）/ Bug 修复（Bug_fix）/ 需求（Demand）/ 内部任务（Internal_task）/ 投诉（Complaint，不毕业）|
| 投诉（Complaint） | 第 5 种分类，纯情绪/服务态度类反馈，**永远不自动毕业**，只能人工关闭或转型为其他类型 |
| owner-split（按责任人拆分） | 一个 hub_issue 的研发工作按责任人拆成多个 Linear 子 issue，各自独立跟踪 + 独立通知客户进度 |
| 反思诊断 | 针对 AI 客服 escalation（客户对 AI 答复不满转人工）的失败原因排查流程 |
| 病因（cause） | 反思诊断得出的失败根因分类：skill（话术没用好）/ knowledge（知识错误过期）/ retrieval（检索缺失）|
| 知识运营（knowledge_op） | 只能用反思诊断工作台（改对客 skill、核对知识库），够不到主管修正权和内部编排 skill 的角色 |
| 内部编排 skill | hub 系统自己流水线用的 Agent 提示词（分类/拆单/查重等），跟「AI 客服对客 skill」是两套独立系统 |
| 三槽版本 | skill 配置的 current（生产）/ draft（候选）/ previous（回滚）三态版本管理模式 |
| 兜底处理人/兜底池 | 路由规则匹配不到任何分工时的默认承接人，防止工单无人认领 |
| SLA | 服务水平承诺（如首次回复时限、解决时限），按产品线配置，超时触发工作台告警 |
