# 知识反哺飞轮 Skill 设计（2026-06-12 草案）

> 来源：D4 第③段开放问题 ④ 定稿——AI 客服知识库在飞书侧，但**FAQ 不能盲目
> 入库**。本 skill 是「去芜存真」的校验飞轮，作为独立重要待办。
> 状态：设计草案，待评审。

## 一、问题：为什么不能直接把人工答复灌进知识库

朴素飞轮（错误版）：`AI 答不上 → hub 人工回复 → 直接进知识库 → 下次 AI 能答`。

它的三个致命缺陷：
1. **错误答复污染**：人工回复也可能是错的/临时的/个例特例，直接入库会让 AI
   学到错知识，错误被放大复用
2. **存量冲突无视**：新答复可能与库里已有条目矛盾（产品迭代后旧答复过时），
   盲目追加 → 库里同一问题多个互相打架的答案
3. **只增不减**：错误来源永远留在库里持续误导，没有剔除/修正机制

**正确目标（用户原话）**：确保 FAQ 经过存量知识库的核对和校验，去芜存真——
**不仅补全正确的版图，也能对所有错误答复来源予以剔除或修正**。

## 二、飞轮核心：候选答复的四象限裁决

每条「候选知识」（来自 hub 人工回复 / escalation 解决方案）入库前，先与存量
知识库做**语义检索召回**（复用现有 embeddings + 余弦，同 dedup 套路），
再由校验 Agent 判定落入四象限之一：

```
                  与存量库的关系
                  ┌────────────────────┬────────────────────┐
                  │   库里无相似条目     │   库里有相似条目      │
     ┌────────────┼────────────────────┼────────────────────┤
     │ 候选答复    │  ① NEW              │  ② REINFORCE        │
     │ 经核对正确  │  补全版图：新增条目  │  增强/合并：提置信、 │
     │            │  （需人工终审）      │  补充措辞，不新增重复 │
     ├────────────┼────────────────────┼────────────────────┤
     │ 候选答复    │  ③ DISCARD          │  ④ CORRECT          │
     │ 经核对存疑/ │  剔除：不入库，      │  修正：标记存量条目  │
     │ 错误        │  记录错误来源        │  过时/错误 →         │
     │            │                     │  退役或替换          │
     └────────────┴────────────────────┴────────────────────┘
```

- **① NEW**：库里没有 → 这是真正的版图补全（用户要的"补全正确版图"）
- **② REINFORCE**：库里有且一致 → 不制造重复，反而强化已有条目
- **③ DISCARD**：候选本身被判错/存疑 → 不入库 + 记录"错误答复来源"供审计
- **④ CORRECT**：候选正确但**与存量条目矛盾** → 说明**存量是错的/过时的** →
  触发对存量条目的退役或修正（用户要的"对错误答复来源予以剔除或修正"）

象限 ④ 是飞轮的灵魂：它让知识库能**自我纠错**，而不只是单向膨胀。

## 三、"经核对正确"如何判定（避免 AI 自说自话）

候选答复的"正确性"不能只靠一个 LLM 拍脑袋。多源信号加权：

| 信号 | 权重 | 来源 |
|------|------|------|
| 人工答复者角色 | 高 | supervisor/admin 的 reply_sync 答复 > 普通 |
| 客户是否验收 | 高 | escalation 后客户确认解决 / hub_issue 进入 verified |
| 与存量高置信条目的一致性 | 中 | ② 象限的一致性本身是正确性证据 |
| 校验 Agent 的事实核对 | 中 | LLM 对照产品文档/已知正确条目判断 |
| 答复时效 | 低 | 产品版本号/时间戳——旧答复对新版本可能失效 |

**关键约束（用户要求）**：象限 ① NEW 和 ④ CORRECT（动到知识库的写操作）
**必须人工终审**，Agent 只产出"提案 + 证据 + 象限判定"，落 `agent_decisions`
审计，人在飞书/hub 工作台一键采纳——与 split/dedup 完全同构的灰度剧本。
②REINFORCE / ③DISCARD 风险低，可配置为自动。

## 四、为什么做成 skill（而非塞进工单链）

- **触发时机多样**：reply_sync 保存后、escalation 解决后、定期批量回扫存量库
  ——不是单一工单事件，skill 形态更合适
- **跨系统**：读飞书知识库 + 读 hub 答复历史 + 写飞书知识库，是编排任务
- **可独立演进**：飞轮的判定规则会持续调优，与工单主链解耦
- 形态参考：类似已有的 Claude skill，输入「候选答复批次」→ 输出「四象限提案 +
  人工终审清单」

## 五、最小数据/接口需求（待 AI 客服+飞书 API 补充后细化）

| 需求 | 用途 | 待确认 |
|------|------|--------|
| 飞书知识库**读** API（检索+取条目）| 召回存量、做一致性核对 | 接口形态/鉴权 |
| 飞书知识库**写** API（新增/更新/退役条目）| 落 ①④ 终审结果 | 是否支持条目级更新与退役 |
| 知识条目的稳定 ID + 版本 | ④ 修正时定位存量条目 | 飞书侧条目是否有稳定标识 |
| hub 候选答复来源 | reply_sync 历史 / escalation 解决方案 | 已有（hub_issue_reply_history）|

可能新增表 `knowledge_review_log`（候选 → 象限 → 终审结果审计），
或复用 `agent_decisions`（decision_type 扩 `knowledge_new/reinforce/discard/correct`）。
**倾向复用 agent_decisions**——与全家桶审计/revert 机制一致，不另起炉灶。

## 六、与第③段主链的接口点

```
escalation 判回 Operation → hub 主管 reply_sync 撰写答复
                                    │
                                    ▼（候选答复产生）
                          知识反哺 skill（本设计）
                                    │ 四象限裁决 + 人工终审
                                    ▼
                          飞书知识库去芜存真
                                    │
                                    ▼
                          AI 客服下次能答（飞轮闭合）
```
