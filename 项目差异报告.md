## 一、一句话定性：两种不同的产品哲学

| | 主项目 ticket-hub | sample/ticket |
|---|---|---|
| **核心定位** | **跨源聚合枢纽 + Agent 辅助 + 主管治理**。Agent 只产「提案」写审计，主管确认，灰度开关逐步放开自动 | **AI 全自动答客户引擎**。AI 直接答客户、自动关单、自动建 Linear、自动回写源系统 |
| **人在环** | 强（agent_decisions 审计 + revert + execute-* 手动 + 灰度开关）| 弱（流水线端到端自动跑，末端才转人工）|
| **AI 答客户** | **不自建**，接飞书侧已有 AI 客服；hub 只管「答不上之后的事」 | **自建完整 AI 答复闭环**（ask agent + answer-router 判 A/B/C/D）|
| **成熟度取向** | 工程治理成熟（611 测试 / 90% 覆盖 / Alembic 12 迁移 / lint+mypy 严格 / CI 门槛）| 业务闭环完整但工程治理薄弱（14 测试 / 无 Alembic / 无 CI）|

**关键洞察**：sample 不是「主项目的旧版本」，而是**对同一业务的另一种实现选择**——它把主项目「刻意延后/外包」的东西（AI 答复、回写源系统、FAQ 飞轮）都自建了，但把主项目「重投入」的东西（测试、迁移、审计治理）做得很轻。两者各有可借鉴处。

## 二、架构与技术栈对照

| 维度 | 主项目 | sample/ticket |

| 编排 | 分散：每个 Agent 独立函数，webhook 串 `run_post_ingest_agents` | **集中 harness**：`pipeline/runner.py` 是唯一读写 DB 的编排层，skill 是纯函数，每步落 `t_skill_log` |

## 三、数据模型对照

| | 主项目 | sample/ticket |
|---|---|---|
| 工单建模 | **单表 `tickets`**（Raw/Parent/Child 三类型）+ `hub_issues`（4 出口类型）+ `agent_decisions`（审计）| **多表拆分**：`t_ticket_info`(主) + `t_ticket_detail` + `t_ticket_tag` + `t_ticket_org`(源系统字段) + `t_ticket_hub`(研发单) + `t_ticket_merge` |
| 主键 | INT autoincrement（ADR-0002）| INT BIGSERIAL + `seq_ticket_no/seq_hub_no/seq_faq_no` 序列 |
| 向量 | `ticket_embeddings` JSON 列 + Python 余弦（ADR-0015）| 同思路：embedding 存 jsonb + 应用层余弦（也明确放弃 pgvector）✅ 英雄所见略同 |
| 审计 | `agent_decisions`（决策+revert）+ `status_history` | `t_skill_log`（每步流水）+ `t_operation_log` + `t_integration_log` |
| 知识库 | 无自建表（接飞书 wiki）| **`t_faq` + `t_faq_review`（FAQ 收录+审核）** ⭐ |
| Skill | 无（prompts 文件）| **`t_skill_md` + `t_skill_md_history`** ⭐ |
| 分工 | `assignment_scopes_module/feature` | `t_module_owner`（产品线+模块→责任人）|
| 派单 | 无专表（路由落 assigned_user）| `t_dispatch_assignee/config/log`（配额派单）|
| SLA | `status_history` + SLAWatcher（内置阈值）| `t_sla_log` + **`t_holiday`（工作日/节假日感知）** ⭐ |

**取向差异**：主项目「单表多类型 + 强约束 CHECK」赌的是简单一致；sample「按读写频率拆 info/detail/org」赌的是热字段查询性能。两者都合理，主项目的单表对当前量级更省心。

## 五、优缺点分析

### sample/ticket 的优点
1. **业务闭环完整**：从接入到答客户、回写、关单、FAQ 收录一条龙，是一个「能跑通端到端」的最小可用产品。
2. **几个主项目延后/没做的东西已实现**（见 §四 ⭐）：回写源系统、hub-dedup、FAQ 飞轮 v1、PII 脱敏、热加载提示词、NL2SQL、节假日 SLA、资料缺失分支。
3. **harness/skill 架构干净**：编排层唯一管 DB，skill 纯函数易测；每步 `t_skill_log` 可观测。
4. **提示词 DB 化 + 版本化 + 页面可编辑**：正好命中「分类规则人工配置 skill」的诉求。
5. **Docker 化部署**更可移植；「去AI化」开关便于降级排障。

### sample/ticket 的缺点
1. **测试极薄**（14 vs 611）、**无 Alembic**（裸 ddl 改表危险）、**无 CI/lint/mypy 门槛**——工程治理不达生产标准。
2. **自动化激进、人在环弱**：AI 直接答客户/关单/建 Linear，缺审计-revert-灰度三件套，线上误答风险高。
3. **直连 Claude（海外 LLM）**：有 PII 出境合规问题（虽有 desensitize 缓解，但主项目刻意避免）。
4. **runner.py 近 450 行 god-file**：端到端可读，但增长后难维护。
5. **多表拆分**增加 join 复杂度；全异步增加心智负担。

### 主项目的优缺点（对照）
- **优**：治理成熟（审计/revert/灰度）、测试迁移 CI 齐全、拆单/Vision/Linear team 路由独有、PII 边界可控。
- **缺**：AI 答复、回写源、FAQ 飞轮、PII 实现、热加载提示词均**延后或未做**——而这些 sample 都有现成参考。

## 六、选择性合并策略（按价值/风险分级）

> 原则：**主项目是生产基线，不动其治理骨架**。只把 sample 的「具体实现」作为参考，移植主项目已规划但未落地的部分，并套上主项目的审计/灰度/测试规范。**不照搬 sample 的自动化激进路线**。

### 第一梯队 — 高价值、低冲突、强烈建议借鉴

| # | 借鉴项 | 落地方式 | 工作量 | 风险 |
|---|---|---|---|---|
| 1 | **PII 脱敏正则**（`skills/desensitize.py`）| 直接移植为主项目 `app/core/pii/` 的可用实现，替换 Protocol 占位；接海外 LLM 前的过渡方案 | 0.5d | 低 |
| 2 | **DB 化版本提示词**（`t_skill_md` + 热加载 + 历史）| 主项目新增 `prompts` 表 + 后台编辑页，classify/dedup/escalation 提示词迁入；正好实现「分类边界规则人工配置 skill」 | 3~4d | 中（需审计+回滚） |
| 3 | **FAQ 飞轮 v1 参考**（`_faq_record` + `faq-review` + 收录前去重）| 作为知识反哺 skill（第④点）的实现蓝本——主项目设计的四象限比它更严，但它的「收录前 0.85 去重 + LLM 审核 approve/reject」可直接复用 | 设计已就绪，实现期参考 | — |
| 4 | **hub 级语义去重**（建 Linear 前 `_hub_dedup`）| 主项目 `linear_push` 前加一步：embedding 召回已有 hub_issue → 命中则关联不新建 Linear，避免重复 issue | 1~2d | 低 |

### 第二梯队 — 有价值、中等投入、择机

| # | 借鉴项 | 落地方式 | 工作量 |
|---|---|---|---|
| 5 | **回写源系统实现**（KSM lock/handle/supply/unlock）| 主项目 D5 `sync_outbox` sender 直接参考 sample 的 `writeback.py` + ksm_client 写方法 | D5 排期内 |
| 6 | **SLA 工作日/节假日感知**（`t_holiday` + workday）| 主项目 SLAWatcher 增强：阈值计算扣除节假日 | 1~2d |
| 7 | **NL2SQL 主管查工单**（assistant + `sql_guard`）| 新增主管「自然语言查工单」功能，`sql_guard`（SELECT-only+LIMIT+角色脱敏）几乎可原样移植 | 2~3d |
| 8 | **资料缺失(C)出口**（supplyKsmOrder 退回客户补资料）| 主项目 hub_issue 出口类型或 ticket 状态增加「待补资料」，escalation 链可判此分支 | 1d |

### 第三梯队 — 仅作架构参考，不直接合并

- **harness/skill 纯函数 + 唯一 DB 编排层**：理念好，但主项目已是分散 Agent 模式且测试完备，重构收益不抵风险——**只在新模块（如飞轮 skill）局部采用这个模式**。
- **info-dedup 答案复用关单**：与主项目「dedup 仅审计 + 外部 AI 答复」哲学冲突，**不建议**——主项目不自建答客户。

### 明确不建议合并

- sample 的**激进自动化**（AI 自动答客户/关单/建 Linear 无主管确认）——违背主项目治理原则。
- sample 的**裸 ddl 改表 / 弱测试 / 直连 Claude** ——工程与合规倒退。
- sample 的**多表拆分 / 全异步 / Vue 前端**——纯技术栈差异，无合并价值。

## 七、落地建议（顺序）

1. **立刻可做**：移植 PII 脱敏正则（#1）→ 补上主项目最尴尬的占位空缺。
2. **配合「人工配置 skill」诉求**：做 DB 化版本提示词（#2）——这是你之前提过的方向，sample 给了现成范式。
3. **飞轮阶段**：把 sample 的 FAQ 收录+审核（#3）作为知识反哺 skill 的实现参考，叠加主项目设计的四象限校验。
4. **D5 阶段**：回写源系统（#5）直接参考 sample 的 writeback。
5. **hub-dedup（#4）** 可在任意 Linear 相关迭代顺手加，立竿见影减少重复 issue。

> 总结：sample 是一面很好的「镜子」——它证明了主项目延后的那些能力可以落地，并提供了具体写法；但它的工程治理和自动化分寸不适合直接搬。**取其实现、弃其激进、套主项目的治理与测试规范**，是最优合并策略。
