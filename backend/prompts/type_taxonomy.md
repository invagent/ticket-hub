> **总原则：Operation 是默认归属。** 客户的疑问、产品使用咨询、已有功能的操作方法
> 都归 Operation，由 AI 客服 agent 直接答（答不了自动转人工）。只有满足下面 Bug_fix
> 或 Demand 的**严格准入条件**时才归它们；两者都不满足 → Operation。

- **Operation** —（**默认兜底类型**）客户使用咨询、配置说明、操作指导、能力咨询，
  以及一切「客户在问问题 / 想知道怎么做 / 想知道能不能做」的工单。例：「数电配置
  流程是什么」「这个权限在哪里设置」「如何申请/下载补丁」「私有化环境如何升级」
  「**能不能控制下载权限**」「**是否支持 X**」「**如何合并开票**」。
  **判定要点：**
  - 「能否 / 是否支持 / 可不可以 X」是**能力咨询 → Operation**（agent 去查系统到底
    支不支持：支持就教操作，不支持才由主管转 Demand）。**不要因为「客户想要某能力」
    就直接判 Demand**——除非客户明确说「希望新增 / 建议增加 / 现在没有、要你们做」。
  - 客户描述现象/状态（「一直在勾选中」「找不到入口」）并问「什么原因 / 如何处理」，
    没贴出报错文案 → Operation。
  - 涉及补丁、升级、部署、初始化等词，只要是客户「问怎么做」→ Operation（别判
    Internal_task）。
- **Bug_fix** —（严格准入）**已有功能**出现明确故障，且有**具体故障证据**：报错弹窗
  文案（「提示系统异常」「网络错误」「服务器异常」）、程序级异常（堆栈、500、plugin
  not found）、功能明确失效（「点了没反应」「导不出来」「数据错乱」）。例：「调用
  发票云接口报 plugin not found」「OCR 识别出来字段为空」「点击批量导出提示系统
  异常」。**只是描述现象、问怎么办、没有报错文案的 → 不判 Bug_fix，归 Operation。**
- **Demand** —（严格准入）客户**明确要求新增系统当前不存在的功能**，或明确提出功能
  改进/字段扩展/流程优化诉求。强信号：「希望增加」「建议支持」「能不能加个 X」
  「现在（系统）不支持，我们想要」。例：「希望发票池支持自定义字段」「能不能加个
  批量导出按钮」。**关键区分：单纯问「是否支持 X / 能不能做 X」是能力咨询（Operation）；
  只有客户表达了「要你们做出这个功能」的诉求才是 Demand。拿不准是咨询还是诉求 →
  先归 Operation，让 agent 尝试解答。**
- **Internal_task** — **我方内部主动发起**的任务（非客户提出），如运营手动触发的
  批处理、组织/数据初始化、我方内部执行的补丁部署等。**判定铁律：任何来自客户的
  留言/提问/咨询都不是 Internal_task**——客户问「如何申请补丁」是 Operation（咨询），
  只有「我方运营要给某租户批量部署补丁」这种无客户提问、纯内部动作才算。从 KSM/
  智齿等客户工单流入的**几乎不可能**是此类，判 Internal_task 前先自问「这是客户在
  问问题，还是我方内部要做事？」，是客户在问 → 归 Operation/Bug_fix/Demand。
- **Complaint** — 客户对**服务本身**的投诉：响应慢、态度、反复不解决、要求赔偿/
  升级处理等**情绪与服务事故**，而非具体产品问题。强信号：「你们处理太慢」
  「投诉」「要求领导介入」「再不解决就退款」。投诉一律转人工，不进自动化流水线。
